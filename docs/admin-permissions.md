# Роли и права доступа

В auth-service есть три роли:

- `USER`: обычный пользователь.
- `ADMIN`: операторская роль для просмотра пользователей, управления их сессиями и блокировками обычных пользователей.
- `SUPER_ADMIN`: системная роль для управления администраторами, блокировками привилегированных пользователей и client applications.

Роли хранятся в `user_roles` и попадают в access token в claim `roles` без префикса `ROLE_`. Spring Security мапит их в authorities с префиксом `ROLE_`.

## Модель ролей

Роли не выдаются автоматически как иерархия в БД. Если пользователь имеет `SUPER_ADMIN`, в БД достаточно хранить только `SUPER_ADMIN`; роль `USER` ему отдельно не нужна.

Обычная регистрация выдает только `USER`. Bootstrap первого привилегированного пользователя выдает только `SUPER_ADMIN`.

## Матрица прав

| Действие | USER | ADMIN | SUPER_ADMIN |
|---|---:|---:|---:|
| Использовать пользовательские auth endpoints | да | да | да |
| Смотреть список пользователей | нет | да | да |
| Смотреть пользователя | нет | да | да |
| Смотреть сессии пользователя | нет | да | да |
| Отозвать одну сессию пользователя | нет | да | да |
| Отозвать все сессии пользователя | нет | да | да |
| Заблокировать/разблокировать обычного пользователя | нет | да | да |
| Заблокировать/разблокировать ADMIN или SUPER_ADMIN | нет | нет | да |
| Смотреть audit log | нет | да | да |
| Смотреть auth client applications | нет | да | да |
| Создать auth client application | нет | нет | да |
| Изменить auth client application | нет | нет | да |
| Включить auth client application | нет | нет | да |
| Отключить auth client application | нет | нет | да |
| Выдать роль ADMIN | нет | нет | да |
| Отозвать роль ADMIN | нет | нет | да |

## Блокировки пользователей

- Бан хранится отдельной записью в `user_bans`, а не статусом пользователя.
- `expiresAt = null` означает постоянный бан, будущий `expiresAt` — временный.
- Причина бана обязательна.
- Одновременно у пользователя может быть только один активный бан.
- Любое управление собственным баном, включая self-ban и self-unban, запрещено.
- Bootstrap `SUPER_ADMIN` имеет постоянный признак `banProtected` и не может быть забанен.
- `ADMIN` может управлять банами только обычных пользователей.
- `SUPER_ADMIN` может управлять банами других привилегированных пользователей, кроме защищенного bootstrap.
- Бан сразу отзывает все refresh-сессии пользователя.
- Login и refresh забаненного пользователя возвращают `403 USER_BANNED`.
- Истечение временного бана не восстанавливает старые сессии; пользователь должен выполнить новый login.

## Auth client applications

`auth_clients` описывает приложения, которым auth-service может выдавать токены. Сейчас это используется для:

- выбора `aud` в access token;
- claim `client_id` в access token;
- TTL access token, ограниченного глобальным максимумом `APP_SECURITY_ACCESS_TOKEN_TTL`;
- TTL refresh token;
- списка разрешенных origins для CORS preflight и привязки браузерных запросов к auth client;
- отключения client application.

Если client application отключен, новые login/register/refresh для него отклоняются. В stateful-режиме auth-service также отклоняет access token с отключенным `client_id` на своих защищенных endpoint'ах.
